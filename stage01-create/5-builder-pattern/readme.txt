建造者模式是一种"创建型设计模式"，用于构建复杂对象，其核心是分离“组件组装工艺”与“整体构建工序”，主要包含4个核心角色：
1. Product（产品）：需多组件组装的复杂对象，比如案例中的“建筑物类”；
2. Builder（建造者接口）：定义产品各组件的构建标准（含多个步骤），是组件构建的规范，对应案例的“施工方接口”；
3. ConcreteBuilder（具体建造者）：实现建造者接口，负责具体组件的组装（但不控制整体构建逻辑），支持多实现（比如案例中的“别墅施工方”“公寓施工方”）；
4. Director（指导者）：持有建造者接口，控制构建的整体工序（统一流程），对应案例的“工程总监类”。

该模式的核心目标是：将“变”与“不变”分离 —— 让“组件组装工艺”（不同产品的差异化实现）由具体建造者负责（实现“工艺多样化”），
“整体构建工序”由指导者统一控制（实现“工序标准化”），最终通过相同的构建流程，产出不同类型的产品。

举例: 房地产建筑（简化处理）
1. 一个建筑物，不管是普通住宅，还是高楼大厦，都需要包含三个必要属性: 地基、墙体、屋顶（需要构建一个Building实体类）
2. 而不同的房地产开发商，建造的建筑也不一样。普通住宅开发商建筑的是住宅所需的地基、墙体、屋顶。高楼大厦的又是另外的！
   但是他们都必须有构建 地基、墙体、屋顶 三个元素的能力！（抽象为三个方法）-- 定义Builder抽象类，里面有三个抽象方法 和 一个交付产品的 build方法
3. 不同的房地产开发商（比如HouseBuilder）去实现这个 Builder 抽象接口，按自己的方式去创建三个属性（无执行顺序），并且交付对应的商品。
4. 此时又会有一个问题，如果开发商先创建了房顶，然后尝试组装房子，那地基什么都没有，最后的产品质量也是难以保障！
   此时需要一个 指导者 ，去统一控制"整体构建工序"，这个执导者并不是特定监控某个开发商，而只是监控每个开发的流程！
